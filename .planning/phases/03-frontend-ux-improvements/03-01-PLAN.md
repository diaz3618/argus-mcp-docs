The exact CSS to insert:

```css
.class-name {
  color: hsla(210, 100%, 66%, 1);
}

.operator {
  color: gray;
}

.variable {
  color: hsla(335, 73%, 53%, 1);
}

.dark .variable {
  color: hsla(335, 76%, 59%, 1);
}

  color: hsla(38, 92%, 50%, 1);
}

  color: hsla(38, 95%, 58%, 1);
}

.atrule {
  color: hsla(335, 73%, 53%, 1);
}

.dark .atrule {
  color: hsla(335, 76%, 59%, 1);
}

.null-keyword,
.null.token {
  color: hsl(0, 0%, 29%);
}

.dark .null-keyword,
.dark .null.token {
  color: hsl(0, 0%, 71%);
}
```

Rationale for each rule (per D-03 and UI-SPEC):
- `.class-name` → blue `hsla(210,100%,66%,1)`: matches `.function` — same semantic tier as Python class declarations. No dark override needed (same in light and dark per existing `.function` pattern).
- `.operator` → `gray`: matches `.punctuation` — operators have the same visual weight as punctuation. No dark override needed.
- `.variable` → pink `hsla(335,73%,53%,1)` / dark `hsla(335,76%,59%,1)`: matches `.keyword` — bash `$VAR` expansions are binding constructs.
- `.important` → amber light `hsla(38,92%,50%,1)` / amber dark `hsla(38,95%,58%,1)`: distinctive new hue for YAML anchors `&anchor`/`*alias` — not used by any other token.
- `.atrule` → pink `hsla(335,73%,53%,1)` / dark `hsla(335,76%,59%,1)`: matches `.keyword` — YAML directives are directive constructs.
- `.null-keyword`, `.null.token` → gray `hsl(0,0%,29%)` light / `hsl(0,0%,71%)` dark: matches `.constant` — JSON null is a special literal value.

Insert location: after line ~263 (end of existing `.dark .constant, .dark .annotation, .dark .boolean` block), before the `span.katex-display` rule at line ~265.
